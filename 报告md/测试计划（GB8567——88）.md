# 测试计划

## 1 引言

### 1.1 编写目的

本测试计划旨在为"GitGet——GitHub Release 应用商店系统"的测试工作提供完整的指导框架，明确测试范围、测试策略、资源安排、测试用例设计准则和验收标准，确保系统质量符合软件需求说明书中的规定。预期的读者包括测试工程师、开发工程师、项目经理及质量保障人员。

### 1.2 背景

- **a. 测试计划所从属的软件系统的名称**：GitGet（GitHub Release 应用商店系统）
- **b. 该开发项目的历史**
  - 项目启动日期：2026年5月14日
  - 需求分析完成：2026年5月18日
  - 设计完成：2026年5月22日
  - 开发完成：2026年6月12日
  - 测试执行：2026年6月12日 - 6月17日
  - 计划发布：2026年6月20日
  - 在执行本测试计划之前必须完成的工作：所有功能模块开发完成并经过开发人员自测

### 1.3 定义

| 术语 | 定义 |
|------|------|
| 单元测试 | 对软件的最小可测试单元（类/方法）进行的测试 |
| 集成测试 | 验证各模块间交互的正确性 |
| 系统测试 | 在完整系统上验证功能和非功能需求 |
| 验收测试 | 确认系统满足业务需求和交付标准 |
| E2E 测试 | 端到端测试，模拟真实用户操作流程 |
| API 测试 | 对 GitHub API 调用封装的测试 |
| UI 测试 | 对 Avalonia UI 界面的自动化测试 |
| xUnit | .NET 生态的单元测试框架 |
| FluentAssertions | .NET 的断言库，提供可读性强的断言语法 |

## 2 计划

### 2.1 软件说明

#### 系统架构概要

GitGet 系统主要由以下模块组成：

```
┌─────────────────────────────────────┐
│        视图层 (Avalonia UI)          │
│  ┌──────┐ ┌──────┐ ┌──────────┐   │
│  │ 首页  │ │ 搜索  │ │ 项目详情  │   │
│  │ 推荐  │ │ 页面  │ │ + Release │   │
│  └──────┘ └──────┘ └──────────┘   │
│  ┌──────────┐ ┌────────────┐       │
│  │ 下载管理  │ │ 个人中心/   │       │
│  │ 面板     │ │ 系统设置    │       │
│  └──────────┘ └────────────┘       │
├─────────────────────────────────────┤
│     ViewModel 层 (MVVM)             │
│  ┌──────────┐ ┌──────────────┐     │
│  │ 推荐引擎  │ │ Asset 匹配器  │     │
│  └──────────┘ └──────────────┘     │
│  ┌──────────┐ ┌──────────────┐     │
│  │ 下载管理器 │ │ 缓存管理器    │     │
│  └──────────┘ └──────────────┘     │
├─────────────────────────────────────┤
│         数据访问层                    │
│  ┌──────────┐ ┌──────────────┐     │
│  │ GitHub   │ │ 本地存储      │     │
│  │ API 客户端│ │ (SQLite/FS)  │     │
│  └──────────┘ └──────────────┘     │
└─────────────────────────────────────┘
```

#### 功能模块与质量指标

| 模块 | 功能描述 | 输入 | 输出 | 质量指标 |
|------|----------|------|------|----------|
| F-01 OAuth 登录 | GitHub 账号授权登录 | 用户点击登录 | 用户信息、Token | 成功率>99% |
| F-02 热门推荐 | 个性化 + 全局热门推荐 | Star 列表 or 默认 | 项目卡片列表 | 加载<3s |
| F-03 项目搜索 | 关键词搜索 GitHub 仓库 | 搜索关键词 | 搜索结果列表 | 响应<2s |
| F-04 项目详情 | 展示仓库 + Release 信息 | 选择项目 | 详情页面 | 加载<3s |
| F-05 智能匹配 | 自动匹配 OS 对应 Asset | 操作系统信息 | 标记推荐 Asset | 准确率>90% |
| F-06 下载管理 | 带进度条的下载管理 | 点击下载 | 下载进度 + 文件 | 进度更新1次/s |
| F-07 个人中心 | 用户信息、Star 列表、下载历史 | 用户查看 | 个人页面 | 正确显示 |
| F-08 系统设置 | 下载路径、主题、语言等 | 用户配置 | 设置保存生效 | 持久化正确 |
| F-09 缓存系统 | 本地数据缓存减少 API 调用 | API 响应数据 | 缓存数据 | 缓存命中率>60% |

## 3 测试设计说明

### 3.1 测试 1：单元测试设计

#### 3.1.1 控制

- 使用 xUnit.net 作为测试运行器，支持 dotnet watch test 自动重跑
- 使用 Moq 框架模拟接口依赖，自定义 HttpMessageHandler 模拟 GitHub API 响应
- 测试命令：`dotnet test`（单次） / `dotnet watch test`（监听模式）
- 覆盖率报告：`dotnet test /p:CollectCoverage=true /p:CoverletOutput=./coverage/`

#### 3.1.2 输入

- 各方法的参数输入（正常值、边界值、异常值）
- Mock 的 GitHub API 响应数据
- 模拟的文件系统操作（通过 System.IO.Abstractions）

#### 3.1.3 输出

- 测试通过/失败结果
- 代码覆盖率报告（HTML 格式，由 Coverlet + ReportGenerator 生成）
- 日志输出（用于调试失败的测试）

#### 3.1.4 过程

1. 编写测试用例（遵循 Arrange-Act-Assert 模式）
2. 配置 Mock 对象（使用 Moq 模拟接口）
3. 运行测试套件（dotnet test）
4. 检查测试结果和覆盖率报告
5. 修复失败的测试和代码缺陷
6. 重复直到测试全部通过且覆盖率达到目标

### 3.2 测试 2：集成测试设计

#### 3.2.1 控制

- 使用 xUnit.net 进行集成测试
- 自定义 DelegatingHandler 作为 API Mock 层
- 测试环境使用 Microsoft.Extensions.DependencyInjection 容器构建完整的依赖图

#### 3.2.2 输入

- 预设的 GitHub API 响应数据（按场景分别预设为 JSON 文件）
- 用户操作序列（通过 ViewModel 方法调用模拟）
- 网络条件模拟（正常、慢速、断网）

#### 3.2.3 输出

- 完整性检查日志
- API 调用链追踪
- 数据一致性验证结果

#### 3.2.4 过程

1. 使用 ServiceCollection 搭建测试 DI 容器
2. 注入 Mock 的 HttpClient（自定义 HttpMessageHandler）
3. 执行预设的测试场景
4. 验证每个步骤的中间状态和最终结果
5. 记录测试结果和发现的缺陷

### 3.3 测试 3：系统测试设计

#### 3.3.1 控制

- 手动测试 + Avalonia Headless 自动化测试
- 缺陷管理：GitHub Issues，标签分类（bug/feature/enhancement）
- 优先级定义：Critical / High / Medium / Low

#### 3.3.2 输入

- 各测试用例中定义的操作步骤
- 真实 GitHub 账号（测试用）
- 各平台的测试机

#### 3.3.3 输出

- 测试结果记录
- 缺陷报告（GitHub Issues）
- 性能测试报告（响应时间、资源消耗）
- 兼容性测试矩阵

#### 3.3.4 过程

1. 搭建各平台测试环境
2. 按功能模块逐一执行测试用例
3. 发现缺陷后记录到 GitHub Issues
4. 开发工程师修复后重新验证
5. 回归测试确保修复不引入新问题
6. 性能测试使用 .NET Stopwatch 类执行
7. 兼容性测试在各平台上执行核心功能用例
