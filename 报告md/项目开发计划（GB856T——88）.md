# 项目开发计划

## 1 引言

### 1.1 编写目的

本项目开发计划旨在对"GitGet——GitHub Release 应用商店系统"的整个开发过程进行全面规划，明确工作内容、人员分工、进度安排和资源分配，确保项目在规定的时间和预算内高质量完成。预期的读者包括项目管理人员、开发团队成员、测试人员及项目评审人员。

### 1.2 背景

- **a. 待开发的软件系统的名称**：GitGet（GitHub Release 应用商店系统）
- **b. 本项目的任务提出者、开发者、用户**
  - 任务提出者：项目发起人/产品经理
  - 开发者：GitGet 开发团队
  - 用户：GitHub 平台的所有开发者及普通用户
- **c. 该软件系统同其他系统或其他机构的基本的相互来往关系**
  - 与 GitHub REST/GraphQL API 对接
  - 与 GitHub OAuth 认证系统对接
  - 与操作系统本地文件系统交互

### 1.3 定义

| 术语 | 定义 |
|------|------|
| GitGet | 本项目的产品名称 |
| Release | GitHub 仓库发布的软件版本包 |
| Asset | Release 中的附件文件 |

| MVP | 最小可行产品（Minimum Viable Product） |

## 2 项目概述

### 2.1 工作内容

本项目开发需要完成以下主要工作：

1. **项目规划与需求分析**：确定项目范围、技术选型、需求文档编制
2. **系统架构设计**：设计整体架构、组件划分、数据流设计
3. **UI/UX 设计**：界面原型设计、交互流程设计、视觉设计
4. **核心功能开发**：
   - GitHub OAuth 登录模块
   - 热门项目推荐模块
   - 项目搜索模块
   - 项目详情与 Release 展示模块
   - 智能 Asset 匹配模块
   - 下载管理模块
   - 用户个人中心和设置模块
5. **测试与质量保障**：单元测试、集成测试、系统测试、验收测试
6. **文档编制**：用户手册、技术文档、部署文档
7. **部署与发布**：打包构建、分发渠道发布

### 3.5 关键问题

| 关键问题 | 风险等级 | 影响 | 应对策略 |
|----------|----------|------|----------|
| GitHub API 速率限制 | 高 | 高并发场景下 API 请求被限流，影响用户体验 | 1. 实现多层缓存策略（SQLite 缓存）<br>2. 认证用户提升配额至 5000次/小时<br>3. 请求队列和退避机制 |
| 跨平台兼容性 | 中 | 在部分 Linux 发行版上可能存在兼容问题 | 1. 优先支持 Ubuntu 20.04+/Fedora 36+<br>2. 使用 Avalonia UI 跨平台抽象层 |

| OAuth 流程安全 | 中 | Token 泄露风险 | 1. Token 加密存储（AES-256-GCM）<br>2. 使用 PKCE 增强 OAuth 流程<br>3. 无长期 Token 存储 |
| 开发周期紧张 | 中 | 38天内完成所有功能压力较大 | 1. 优先开发高优先级功能（F-01 至 F-06）<br>2. 中优先级功能视进度酌情调整范围 |
| .NET 自包含发布体积 | 低 | 安装包体积较大（~50-80MB） | 1. 使用 ReadyToRun 预编译优化<br>2. 使用 .NET 自包含发布裁剪 |

## 5 专题计划要点

### 5.1 配置管理计划

- 使用 Git 进行版本控制，GitHub 作为远程仓库
- 分支策略：`main`（生产）→ `develop`（开发）→ `feature/*`（功能分支）
- 代码评审：所有 PR 需至少一名其他开发者评审后合并
- 版本号规范：遵循语义化版本（SemVer）规范

### 5.2 质量保证计划

- 编码规范：遵循 .NET 命名规范和 StyleCop 规则
- 单元测试：xUnit + Moq + FluentAssertions，核心功能覆盖 ≥ 60%
- 集成测试：使用自定义 HttpMessageHandler 模拟 GitHub API
- 代码审查：每周代码审查会议
- 持续集成：GitHub Actions 运行 `dotnet test`

### 5.3 测试计划

详见《测试计划（GB8567——88）》。测试分为以下阶段：
1. 单元测试（开发过程中持续进行）
2. 集成测试（Sprint 完成后）
3. 系统测试（全部功能完成后）
4. 验收测试（发布前）

### 5.4 用户培训计划

- 提供中文和英文的用户手册
- 首次启动时展示引导教程
- 建立 FAQ 文档

### 5.5 部署与发布计划

- 构建工具：`dotnet publish` + Avalonia 原生打包
- 分发渠道：
  - GitHub Releases（主要）
  - 独立安装包下载
- 自动更新：内置更新检测机制
- 平台打包格式：
  - Windows：NSIS 安装包 (.exe) / 便携版 (.zip)
  - macOS：DMG (.dmg)
  - Linux：AppImage / deb / rpm

### 5.6 风险应对计划

详见 3.5 节关键问题中的应对策略。此外：

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|----------|
| 团队成员变动 | 低 | 高 | 文档齐全、知识共享、关键模块双人熟悉 |
| 需求变更 | 中 | 中 | 敏捷迭代、变更控制流程、优先级管理 |
| GitHub API 变更 | 中 | 高 | 关注 GitHub Changelog、抽象 API 层、快速适配 |
| 第三方库漏洞 | 低 | 中 | 依赖版本锁定、定期安全检查、Dependabot 集成 |
