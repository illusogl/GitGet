# 概要设计说明书

## 1 引言

### 1.1 编写目的

本概要设计说明书旨在对"GitGet——GitHub Release 应用商店系统"进行系统级的总体设计，明确系统的整体架构、模块划分、接口定义和数据结构设计，为后续的详细设计和编码实现提供总体框架和指导。预期的读者包括系统架构师、开发工程师、测试工程师及项目管理人员。

### 1.2 背景

- **a. 待开发软件系统的名称**：GitGet（GitHub Release 应用商店系统）
- **b. 本项目的任务提出者、开发者、用户**
  - 任务提出者：项目发起人/产品经理
  - 开发者：GitGet 开发团队
  - 用户：GitHub 平台的所有开发者及普通用户
  - 运行环境：用户本地桌面操作系统（Windows/macOS/Linux）

## 2 总体设计

### 2.1 需求规定

#### 功能需求（F-01 至 F-09）

| 编号 | 功能名称 | 输入 | 处理 | 输出 | 性能要求 |
|------|----------|------|------|------|----------|
| F-01 | GitHub OAuth 登录 | 用户点击登录按钮 | OAuth 授权码交换、Token 获取与存储 | 用户头像、用户名、登录状态 | 授权回调后 3s 内完成 |
| F-02 | 热门项目推荐 | 用户 Star 列表或默认空 | 兴趣分类权重计算 + GitHub Search API 调用 | 项目卡片列表 | 加载时间 ≤ 3s |

| F-09 | 离线缓存 | API 响应数据 | 序列化存储到 SQLite | 缓存命中，减少 API 调用 | 命中率 ≥ 60% |

### 2.3 基本设计概念和处理流程

#### 设计理念

1. **分层架构**：采用经典的三层架构（表示层、业务逻辑层、数据访问层），各层职责清晰，便于维护和扩展
2. **MVVM 模式**：UI 层采用 Model-View-ViewModel，通过数据绑定实现视图与逻辑的分离
3. **模块化**：按功能划分模块，每个模块高内聚、低耦合
4. **离线优先**：优先使用本地缓存数据，减少 API 调用以应对 GitHub 速率限制
5. **异步编程**：所有 IO 操作使用 async/await，保证 UI 不阻塞

### 2.6 人工处理过程

1. **GitHub OAuth 首次授权**：用户首次登录时需手动在浏览器中完成 GitHub 授权确认
2. **下载路径选择**：用户首次设置下载目录时需手动选择文件系统路径
3. **应用更新确认**：当检测到新版本时，用户需手动确认是否更新
4. **代理配置**：如需要使用网络代理，用户需手动填写代理地址和端口

### 2.7 尚未解决的问题

1. **GitHub OAuth PKCE 流程**：对于桌面应用，OAuth 回调 URL 需要注册为 `http://127.0.0.1:{port}/callback`，需确定端口冲突处理和注册方式
2. **断点续传兼容性**：部分 CDN 或自定义 Release 托管服务可能不支持 `Range` 请求头，需制定降级方案
3. **Linux 发行版覆盖范围**：Linux 平台版本碎片化严重，最小覆盖范围需进一步明确（当前计划 Ubuntu + Fedora 主力版本）
4. **多账号支持**：当前设计每个设备仅支持登录一个 GitHub 账号，多账号切换为未来扩展项

|------|------|

| Token 端点 | `https://github.com/login/oauth/access_token` |
| 授权类型 | `code`（授权码模式） |
| 客户端 ID | 在 GitHub OAuth App 设置中注册 |
| 回调 URL | `http://127.0.0.1:{port}/callback` |
| 授权范围 | `read:user`, `user:email`, `repo` |

#### 3.2.3 操作系统接口

| 接口 | 用途 | 实现方式 |
|------|------|----------|
| 文件系统读写 | 下载文件存储、配置文件读写 | `System.IO`（.NET 内置） |
| 系统通知 | 下载完成/错误通知 | Avalonia Notification / 平台原生 API |
| 系统托盘 | 后台运行支持 | Avalonia TrayIcon |
| 平台/架构检测 | 获取 OS 类型和 CPU 架构 | `System.Runtime.InteropServices.RuntimeInformation` |
| 默认下载目录 | 获取系统下载文件夹路径 | `Environment.GetFolderPath(Environment.SpecialFolder.UserProfile)` + `"Downloads"` |

### 3.3 内部接口

#### 3.3.1 表示层 ↔ 业务逻辑层

采用 MVVM 模式，View 通过数据绑定与 ViewModel 通信，ViewModel 通过依赖注入调用 Service：

### 4.2 运行控制

| 控制方式 | 操作步骤 |
|----------|----------|
| **启动** | 用户双击应用图标 → 加载 Avalonia 运行时 → 创建主窗口 → 初始化 DI 容器 → 加载首页 → 检查 Token 有效性 → 加载推荐 |

## 6 系统出错处理设计

### 6.1 出错信息

| 错误情况 | 错误码 | 用户提示信息 | 系统处理 |
|----------|--------|-------------|----------|
| 网络不可用 | ErrNetwork | "网络连接不可用，请检查网络设置" | 显示离线提示，使用缓存数据 |
| GitHub API 限流 | ErrRateLimit | "API 请求过于频繁，请在 X 分钟后重试" | 显示剩余配额，自动退避重试 |
| Token 无效/过期 | ErrTokenInvalid | "登录已过期，请重新登录" | 尝试刷新 Token，失败则提示重新登录 |
| 文件写入失败 | ErrFileWrite | "无法写入文件，请检查磁盘空间和权限" | 提示用户选择其他路径 |
| 搜索无结果 | ErrSearchEmpty | "未找到匹配的项目" | 显示空结果页 |
| 下载链接失效 | ErrDownloadFail | "下载链接无效，文件可能已被删除" | 尝试重新获取 Asset 信息 |
| 应用崩溃 | ErrCrash | "应用出现异常，正在恢复..." | 自动重启，恢复未完成下载 |

### 6.2 补救措施

#### a. 后备技术

| 场景 | 后备措施 |
|------|----------|
| 本地缓存损坏/丢失 | 清空缓存数据库，重新请求 API |
| Token 文件损坏 | 提示用户重新登录 |
| 数据库损坏 | 重建数据库文件，从备份恢复（如有） |
| 下载文件损坏 | 检测完整性（如 Content-Length 对比），重新下载 |

#### b. 降效技术

| 场景 | 降效方案 |
|------|----------|
| GitHub API 不可用 | 完全依赖本地缓存展示已有数据，提示用户稍后重试 |
| 未登录状态 | 推荐功能降级为全局 Trending，搜索功能正常 |
| 下载服务不支持续传 | 降级为从头重新下载 |
| 系统通知不可用 | 应用内通知替代系统通知 |

#### c. 恢复及再启动技术

| 场景 | 恢复策略 |
|------|----------|
| 应用异常退出 | 启动时扫描 `download_tasks` 表中状态为 `downloading` 的任务，尝试断点续传 |
| 下载中断 | 保留已下载的临时文件（`.gitget.part`），下次启动时检查并尝试续传 |
| OAuth 流程中断 | 用户可重新点击登录重新开始 |
| 数据库操作失败 | 回滚事务，记录错误日志 |

### 6.3 系统维护设计

| 维护项目 | 方式 | 频率 |
|----------|------|------|
| 缓存清理 | 自动清理过期缓存，设置中提供"清除缓存"按钮 | 自动按有效期清理 / 手动 |
| 数据库优化 | `VACUUM` 命令压缩数据库 | 每月 / 手动 |
| 日志记录 | 错误日志写入 `AppData/GitGet/logs/error.log` | 持续记录 |
| 应用更新 | 内置自动更新检测 | 每次启动 |
| 健康检查 | 启动时自检（Token 有效性、数据库完整性、网络连通性） | 每次启动 |
