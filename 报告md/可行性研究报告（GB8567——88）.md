# 可行性研究报告

## 1 引言

### 1.1 编写目的

本可行性研究报告旨在对"GitGet——GitHub Release 应用商店系统"项目从技术、经济、社会和法律等多个维度进行全面分析，评估项目是否具备启动条件。预期的读者包括项目决策者、投资方、开发团队及项目管理人员。

### 1.2 背景

- **A. 所建议开发的软件系统的名称**：GitGet（GitHub Release 应用商店系统）
- **B. 本项目的任务提出者、开发者、用户**
  - 任务提出者：项目发起人/产品经理
  - 开发者：GitGet 开发团队
  - 用户：GitHub 平台的所有开发者及普通用户，特别是需要便捷下载和管理 GitHub 项目 Release 资源的用户
- **C. 该软件系统同其他系统或其他机构的基本的相互来往关系**
  - 与 GitHub REST API v3 / GraphQL API v4 对接，获取仓库信息、Release 资源、用户数据
  - 与 GitHub OAuth 认证系统对接，实现用户登录
  - 可能集成第三方包管理工具（如 Winget、Scoop、Homebrew）以提供系统级安装支持

### 1.3 定义

| 术语 | 定义 |
|------|------|
| GitGet | 本项目名称，GitHub Release 应用商店系统 |
| Release | GitHub 仓库中发布的软件版本包，包含可下载的二进制文件 |
| Asset | Release 中的附件资源（如 .exe、.dmg、.deb、.zip 等） |
| OAuth | 开放授权协议，用于用户身份认证 |
| Star | GitHub 用户对仓库的收藏标记 |
| API | 应用程序编程接口 |

## 2 可行性研究的前提

### 2.1 要求

**A. 功能**
1. **热门项目推荐**：基于用户已 Star 项目的类别偏好，推荐当前热门仓库及其 Release
2. **项目搜索**：允许用户按关键词搜索 GitHub 中的公开仓库
3. **项目详情与 Release 展示**：点进项目后自动检测当前操作系统，展示匹配的 Release Asset
4. **带进度条的下载**：提供可视化的下载进度条，支持暂停/继续/取消
5. **GitHub 账号登录**：支持通过 GitHub OAuth 进行用户身份认证

**B. 性能**
- 页面首次加载时间不超过 3 秒（在常规网络条件下）
- 搜索响应时间不超过 2 秒
- 下载速度受限于用户网络带宽，客户端下载管理开销不超过 CPU 的 5%

**C. 输出**
- 项目列表（含名称、描述、Star 数、最近更新时间）
- Release 列表（含版本号、发布时间、下载链接）
- 下载进度信息（实时百分比、速度、预计剩余时间）
- 推荐项目列表（个性化推荐结果）

**D. 输入**
- 搜索关键词（文本）
- GitHub OAuth 凭据
- 操作系统类型（自动检测）
- 用户的 Star 数据（通过 GitHub API 获取）

**E. 处理流程和数据流程**
1. 用户登录 → GitHub OAuth 授权 → 获取用户信息及 Star 列表
2. 分析 Star 项目 → 提取语言/标签分类 → 推荐同类别热门项目
3. 用户搜索 → 调用 GitHub Search API → 展示结果列表
4. 用户选择项目 → 获取 Releases → 匹配当前 OS → 展示下载链接
5. 用户点击下载 → 创建下载任务 → 实时更新进度 → 完成下载

**F. 安全与保密要求**
- 用户访问令牌（Token）必须加密存储
- 所有 API 通信使用 HTTPS
- 不存储用户的 GitHub 密码
- 用户数据仅用于功能实现，不上传至第三方

**G. 同本系统相连接的其他系统**
- GitHub API（REST + GraphQL）
- GitHub OAuth 认证服务器

**H. 完成期限**
2026年6月20日

### 2.2 目标

- **A.** 减少用户在 GitHub 上查找和下载软件 Release 的流程复杂度
- **B.** 通过个性化推荐提高用户发现高质量开源软件的效率
- **C.** 提供类似应用商店的直观体验，降低非技术用户的使用门槛
- **D.** 实现跨平台 Release 的智能匹配，避免用户下载错误版本
- **E.** 建立统一的下载管理入口，提升下载体验

### 2.3 条件、假定和限制

| 项目 | 内容 |
|------|------|
| a. 系统运行寿命 | 至少 1 年 |
| b. 方案选择时间 | 2 周内完成技术选型 |
| c. 经费来源 | 自筹资金 |
| d. 法律政策限制 | 需遵守 GitHub API 使用条款及速率限制 |
| e. 软硬件环境限制 | 支持 Windows/macOS/Linux 主流版本 |
| f. 可利用资源 | GitHub 公开 API、开源组件、现有技术栈 |
| g. 最晚上线时间 | 2026年6月20日 |

### 2.4 进行可行性研究的方法

采用以下方法进行评估：
1. **市场调研**：分析竞品（如 GitHub Releases 原生页面、Chocolatey、Scoop 等包管理器）
2. **技术原型验证**：构建最小可行原型验证核心功能
3. **成本估算**：按工作量法估算人力、设备、运营成本
4. **风险评估**：识别关键技术风险并制定应对策略

### 2.5 评价尺度

| 维度 | 权重 | 评价指标 |
|------|------|----------|
| 功能完整性 | 40% | 核心需求覆盖度 |
| 开发周期 | 25% | 是否能在截止日期前完成 |
| 用户体验 | 20% | 易用性、流畅度 |
| 可维护性 | 15% | 代码规范、文档完整度 |

## 3 对现有系统的分析

### 3.1 处理流程和数据流程

**现有方式（直接在 GitHub 上操作）：**
1. 用户打开 GitHub 网站 → 搜索项目
2. 进入项目页面 → 点击 Releases 标签
3. 手动浏览 Assets → 根据文件名判断是否匹配操作系统
4. 点击下载 → 浏览器默认下载管理（无进度显示或功能有限）

### 3.2 工作负荷

- 用户平均需要 3-5 次点击才能找到匹配的 Release 文件
- 搜索热门项目依赖第三方网站（如 GitHub Trending）或手动发现
- 无个性化推荐机制，用户需自行跟踪关注的项目更新

### 3.3 费用开支

现有模式无直接费用，但存在以下隐性成本：
- 用户时间成本高
- 非技术用户容易下载错误版本的 Asset
- 多平台用户需要手动辨别不同操作系统对应的文件

### 3.4 人员

现有系统由 GitHub 平台原生提供，无需额外人员维护。但本项目需组建开发团队：
- C# .NET 桌面开发工程师 1 人
- UI/UX 设计师 1 人
- 测试工程师 1 人

### 3.5 设备

- 开发用计算机（Windows/macOS/Linux 各至少一台用于兼容性测试）
- 无服务器成本（桌面应用直接运行在用户本地）

### 3.6 局限性

1. **无个性化推荐**：GitHub 原生缺少基于用户偏好的个性化推荐
2. **无系统匹配**：用户需手动判断 Asset 是否适配当前操作系统
3. **下载体验差**：浏览器下载管理器功能单一，不支持自动重试、断点续传
4. **发现效率低**：好的项目难以被发现，依赖外部平台（如 Hacker News、Reddit）
5. **非统一入口**：每个项目的 Release 页面风格一致但分散，缺乏聚合体验

## 4 所建议的系统

### 4.1 对所建议系统的说明

GitGet 是一个专注于 GitHub Release 资源发现与下载的应用商店系统。它通过以下方式满足 2.1 节列出的要求：

- **个性化推荐**：利用用户的 Star 数据提取兴趣标签，结合 GitHub Trending 算法，推荐相关热门项目
- **智能搜索**：封装 GitHub Search API，提供高效的仓库搜索体验
- **智能匹配**：自动检测用户操作系统，仅展示兼容的 Release Asset
- **高级下载管理**：基于 .NET 流式下载引擎，提供实时进度、暂停/续传等功能
- **GitHub OAuth 登录**：安全、标准化的认证流程

### 4.2 处理流程和数据流程

```
[用户] → [GitGet 桌面客户端 (Avalonia UI)]
               ↓
         [OAuth 登录] ←→ [GitHub OAuth API]
               ↓
         [获取 Star 列表 / 搜索仓库 / 获取 Release]
               ↓
         [GitHub REST API]
               ↓
         [SQLite 本地缓存]
               ↓
         [智能匹配操作系统 → 显示可用 Asset]
               ↓
         [用户点击下载 → 下载管理器 (.NET HttpClient)]
               ↓
         [实时进度显示 → 完成]
```

### 4.3 改进之处

| 目标 | 现有系统 | 改进后的 GitGet |
|------|----------|-----------------|
| 项目发现 | 无个性化 | 基于 Star 类别的热门推荐 |
| 搜索效率 | 通用搜索 | 针对 Release 资源优化 |
| 系统匹配 | 手动辨别 | 自动匹配，只显示兼容版本 |
| 下载体验 | 基础下载 | 进度条、暂停、续传、重试 |
| 用户系统 | 无 | GitHub OAuth 统一认证 |

### 4.4 影响

#### 4.4.1 对设备的影响

- 桌面用户无需额外硬件，只需安装 GitGet 客户端
- 无需服务器成本，所有处理在用户本地完成

#### 4.4.2 对软件的影响

- 客户端需适配主流操作系统（Windows 10+/macOS 11+/Ubuntu 20.04+）
- 使用 .NET 9 运行时（可随应用分发）
- 依赖的开源库：Avalonia UI（桌面 UI 框架）、Microsoft.Data.Sqlite（数据库）、CommunityToolkit.Mvvm（MVVM 框架）

#### 4.4.3 对用户单位机构的影响

- 面向个人开发者及普通用户，无需机构性调整
- 对团队用户而言，可帮助统一内部使用的开源工具版本管理

#### 4.4.4 对系统运行过程的影响

- **用户操作流程**：登录 → 浏览推荐/搜索 → 选择项目 → 选择版本 → 下载
- **数据管理**：本地 SQLite 缓存用户搜索历史和偏好设置
- **故障处理**：网络中断时下载自动暂停并在恢复后继续
- **数据安全**：Token 使用 AES-256-GCM 加密存储

#### 4.4.5 对开发的影响

- 开发者需熟悉 GitHub API 速率限制及分页策略
- 需要实现 OAuth 流程的完整前后端交互
- 需要处理多平台 Asset 匹配逻辑

#### 4.4.6 对地点和设施的影响

- 开发团队可在远程协作，通过 Git 进行版本管理
- 需要稳定的网络环境以频繁调用 GitHub API

#### 4.4.7 对经费开支的影响

详见第 6 章投资及效益分析。

### 4.5 局限性

1. **依赖 GitHub API 速率限制**：未经认证的 API 调用每小时仅 60 次，认证后为 5000 次/小时，高频使用需优化缓存策略
2. **仅支持公开仓库**：私有仓库的 Release 仅对拥有权限的用户可见且需额外处理
3. **Release 格式多样性**：部分项目的 Asset 命名不规范，自动匹配可能存在误差
4. **非实时更新**：GitHub Trending 数据存在延迟，推荐结果可能不是绝对最新

### 4.6 技术条件方面的可行性

| 条件 | 评估 |
|------|------|
| a. 功能目标能否达到 | 在规定的约束下，所有核心功能均可实现 |
| b. 现有技术能否实现 | 所有依赖技术均已成熟（.NET 9、Avalonia UI、GitHub API、OAuth 2.0） |
| c. 开发人员要求 | 需具备 C# .NET 桌面开发能力，熟悉 GitHub API，团队可通过现有成员满足 |
| d. 规定期限内能否完成 | 项目范围合理，按计划执行可以在 6 月 20 日前完成 |

**结论：技术条件完全可行。**

## 5 可选择的其他系统方案

### 5.1 可选择的系统方案 1：纯 Web 应用（SaaS 模式）

- **描述**：完全基于浏览器的 Web 应用，后端服务器代理 GitHub API 请求
- **优点**：跨平台无需安装，更新即时
- **缺点**：需要维护服务器，下载受浏览器限制（无法实现高级下载管理功能）
- **未被选中的理由**：失去下载管理的核心优势，失去了本地应用的操作流畅度

### 5.2 可选择的系统方案 2：Electron + JS 前端

- **描述**：使用 Electron 封装 Web 前端 + Node.js 后端
- **优点**：前端生态丰富，UI 定制灵活
- **缺点**：安装包体积大（~100MB+），内存占用高，需要同时维护 JS 和 C# 两套代码
- **未被选中的理由**：增加技术栈复杂度，6月20日时间紧张

### 5.3 选中的系统方案：C# 全栈桌面应用

- **描述**：使用 C# + .NET 9 + Avalonia UI 构建跨平台桌面应用
- **选中的理由**：
  - 单一语言完成全栈开发（UI + 业务逻辑 + 数据访问）
  - C# 的 HttpClient 和流式下载天然适合下载管理器
  - Avalonia UI 一套代码运行 Windows/macOS/Linux
  - 团队聚焦 C#，开发效率最高

## 7 社会因素方面的可行性

### 7.1 法律方面的可行性

- **GitHub API 使用条款**：本项目严格遵循 GitHub API 使用条款，允许使用公开数据
- **知识产权**：不存储或分发未经授权的代码，仅提供 Release 下载链接
- **用户数据**：仅存储用户授权范围内的公开信息，符合数据保护相关法规
- **开源协议**：项目本身将采用 MIT 协议开源

### 7.2 使用方面的可行性

- **目标用户**：GitHub 平台拥有超过 1 亿开发者，目标用户群体庞大
- **使用门槛**：界面友好，符合应用商店操作习惯，非技术用户也可使用
- **技术能力**：用户只需基本的计算机操作能力
- **预期使用频度**：日常发现和下载开源软件时使用，中等使用频度
