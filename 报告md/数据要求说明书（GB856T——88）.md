# 数据要求说明书

## 1 引言

### 1.1 编写目的

本数据要求说明书旨在对"GitGet——GitHub Release 应用商店系统"所涉及的所有数据元素进行全面的逻辑描述和数据采集说明，为数据库设计、数据接口设计和数据处理流程设计提供依据。预期的读者包括系统架构师、数据库设计师、C# 后端开发工程师及测试工程师。

### 1.2 背景

- **a. 待开发软件系统的名称**：GitGet（GitHub Release 应用商店系统）
- **b. 本项目的任务提出者、开发者、用户**
  - 任务提出者：项目发起人/产品经理
  - 开发者：GitGet 开发团队
  - 用户：GitHub 平台的所有开发者及普通用户
  - 运行环境：用户本地桌面操作系统（Windows/macOS/Linux）

### 1.3 定义

| 术语 | 定义 |
|------|------|
| 静态数据 | 系统运行过程中主要作为参考、不随运行而改变的数据 |
| 动态数据 | 运行中需要输入、输出或发生变化的动态数据 |
| 内部生成数据 | 系统运行过程中内部产生的中间数据或缓存数据 |
| Repository | GitHub 仓库，包含代码、Issues、Release 等资源 |
| Release Asset | Release 中的附件二进制文件 |
| Token | OAuth 2.0 访问令牌 |

### 1.4 参考资料

1. GB856T——88 《计算机软件产品开发文件编制指南》

### 2.5 数据约定

#### 容量限制

| 数据项 | 最大值 | 说明 |
|--------|--------|------|
| 本地缓存数据库大小 | 100 MB | SQLite 数据库文件大小限制 |
| 下载任务队列长度 | 100 | 同时最多 100 个下载任务（含排队） |
| 并发下载数 | 10 | 用户可配置，最大 10 |
| Token 存储 | 1 条 | 每设备仅存储一个用户的 Token |
| 设置项数量 | 约 50 个 | JSON 配置文件中的键值对数量 |

#### 数据格式约定

1. **时间格式**：统一使用 ISO 8601 格式（如 "2026-05-14T09:30:00.000Z"），所有时间存储为 UTC
2. **数值精度**：文件大小以字节为单位存储，展示时自动换算为 KB/MB/GB
3. **字符编码**：所有字符串数据使用 UTF-8 编码
4. **Token 存储**：使用 AES-256-GCM 加密后存储在本地 SQLite 数据库中
5. **缓存键格式**：`cache:{type}:{key}`，如 `cache:repo:microsoft/vscode`

#### 临界性限制

- Token 过期后必须及时刷新，否则核心功能（Star 获取、个性化推荐）不可用
- API 速率限制（未认证 60 次/小时，认证后 5000 次/小时）为硬性限制，必须设计缓存和配额管理策略
- 未登录状态下推荐功能降级为全局 Trending，搜索功能正常

## 3 数据的采集

### 3.1 要求和范围

#### 3.1.1 输入数据的来源

| 数据项 | 来源 | 采集方式 |
|--------|------|----------|
| 仓库数据 | GitHub REST API | 通过 GitHub Search/Repositories 和 /repos/{owner}/{repo} 接口获取 |
| Release 数据 | GitHub REST API | 通过 /repos/{owner}/{repo}/releases 接口获取 |
| Asset 数据 | GitHub REST API | 通过 Release 响应中的 assets 数组获取 |
| 用户公开数据 | GitHub REST API | 通过 /user 或 /users/{username} 接口获取（需 Token） |
| 用户 Star 列表 | GitHub REST/GraphQL API | 通过 /users/{username}/starred 或 GraphQL 查询获取 |
| 操作系统信息 | .NET RuntimeInformation | 通过 System.Runtime.InteropServices.RuntimeInformation 获取 |
| OAuth Token | GitHub OAuth API | 通过 OAuth 授权码交换获取 |

#### 3.1.2 数据输入媒体和硬件

- 用户键盘和鼠标：输入搜索关键词、选择设置
- 网络接口：所有 GitHub API 数据通过 HTTPS 协议获取
- 浏览器（OAuth 流程）：用户通过系统默认浏览器完成 GitHub 授权

#### 3.1.3 输出数据的接受者

| 数据项 | 接受者 | 交付方式 |
|--------|--------|----------|
| 项目卡片列表 | 最终用户 | Avalonia UI 屏幕渲染 |
| Release 信息 | 最终用户 | Avalonia UI 屏幕渲染 |
| 下载进度 | 最终用户 | 进度条实时更新 |
| 下载文件 | 最终用户 | 保存到本地磁盘 |
| 通知消息 | 最终用户 | 系统通知 |

#### 3.1.4 输出数据的形式和设备

| 输出类型 | 形式 | 设备 |
|----------|------|------|
| 项目展示 | 富文本 UI 卡片（Avalonia ItemsControl） | 主应用窗口 |
| Release 列表 | 结构化列表 UI（Avalonia DataGrid） | 主应用窗口 |
| 下载进度 | 可视化进度条（ProgressBar）+ 数值 | 下载管理面板 |
| 下载文件 | 原始二进制文件 | 本地磁盘 |
| 系统通知 | 文本 + 图标 | 操作系统通知栏 |

#### 3.1.5 数据值的范围

| 字段 | 最小值 | 最大值 | 单位 |
|------|--------|--------|------|
| Star 数 | 0 | 无上限 | 个 |
| 文件大小 | 0 | 无上限 | 字节 |
| 并发下载数 | 1 | 10 | 个 |
| 搜索关键词长度 | 1 | 200 | 字符 |
| 缓存有效期 | 1 | 1440 | 分钟 |

#### 3.1.6 量纲

- **文件大小**：以字节（Byte）为基本单位，显示时自动换算为 KB（1024 字节）、MB（1024² 字节）、GB（1024³ 字节）
- **下载速度**：以 字节/秒（Bps）为基本单位，显示时自动换算为 KB/s 或 MB/s
- **时间**：所有存储使用毫秒级 Unix 时间戳（UTC），展示时转换为本地时间
- **进度百分比**：0.0% - 100.0%，精度 0.1%

#### 3.1.7 更新和处理频度

| 数据项 | 更新频度 | 触发条件 |
|--------|----------|----------|
| 推荐项目列表 | 每次访问首页 | 用户访问首页时 |
| 搜索结果 | 每次搜索 | 用户提交搜索时 |
| 仓库详情 | 每次点击项目 | 用户进入项目详情页时 |
| Release 列表 | 每次查看 | 用户查看项目 Releases 时 |
| 下载进度 | 每秒 1 次 | 下载任务进行中 |
| Star 列表缓存 | 每 6 小时 | 定时刷新 + 手动刷新 |
| 用户兴趣模型 | 登录后 + 每 6 小时 | 登录事件 + 定时刷新 |

### 3.2 输入的承担者

- **搜索输入**：最终用户通过应用界面输入
- **OAuth 授权**：用户通过浏览器完成，由 GitHub OAuth 系统处理
- **设置输入**：最终用户通过设置界面配置
- **操作系统信息**：由 .NET RuntimeInformation 自动检测，无需用户输入
- **GitHub API 数据**：由应用通过 HttpClient 自动调用 API 获取，无需用户手动操作

### 3.3 预处理

#### 3.3.1 数据格式转换

- **时间数据**：将 GitHub API 返回的 ISO 8601 字符串解析为本地 DateTime 对象
- **文件大小**：将字节数转换为显示友好的单位（KB/MB/GB）
- **Markdown 渲染**：将 Release body 的 Markdown 内容通过 Markdown 库渲染为富文本
- **Asset 匹配**：根据文件名和操作系统类型，应用匹配规则确定推荐的 Asset

#### 3.3.2 数据缓存预处理

| 步骤 | 说明 |
|------|------|
| 1. 序列化 | 将 API 返回的 JSON 数据通过 System.Text.Json 序列化为持久化格式 |
| 2. 去重 | 移除重复数据记录 |
| 3. 索引 | 按仓库名、标签等字段创建 SQLite 索引加速查询 |
| 4. 压缩 | 对较大的 body 文本进行压缩存储 |
| 5. 加密 | Token 数据使用 System.Security.Cryptography (AES-256-GCM) 加密后存储 |

#### 3.3.3 下载任务预处理

| 步骤 | 说明 |
|------|------|
| 1. URL 验证 | 验证下载 URL 格式有效性（Uri.TryCreate） |
| 2. 文件名处理 | 处理文件名中的非法字符（Path.GetInvalidFileNameChars） |
| 3. 冲突检查 | 检查目标文件是否已存在，生成唯一名 |
| 4. 磁盘空间检查 | 预估所需空间，检查目标磁盘可用空间（DriveInfo.AvailableFreeSpace） |
| 5. 权限检查 | 验证目标目录的可写权限 |

### 3.4 影响

#### 对设备的影响

- 系统需要维护一个本地 SQLite 数据库，占用约 50-100MB 磁盘空间
- 下载文件占用用户指定的磁盘空间，由用户自行管理
- 内存占用：正常运行约 100-200MB，下载任务进行中额外增加约 50MB/下载任务

#### 对软件的影响

- 需要使用 .NET HttpClient 的流式下载 API 支持进度回调
- 需要使用 System.Security.Cryptography 进行 Token 安全存储
- 需要实现缓存管理机制以遵守 API 速率限制
- 需要跨平台的文件系统操作兼容性处理（Avalonia UI 已抽象）

#### 对用户的影响

- 用户需了解基本的下载目录管理
- OAuth 授权流程需要用户浏览器支持
- 用户可自主清除缓存数据释放空间

#### 对开发单位的影响

- 数据采集模块需设计为可测试的接口封装
- 需建立 API 模拟测试数据以支持离线开发
- 需实现数据迁移机制以支持未来数据格式变更
