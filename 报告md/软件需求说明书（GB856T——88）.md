# 软件需求说明书

## 1 引言

### 1.1 编写目的

本软件需求说明书旨在详细定义"GitGet——GitHub Release 应用商店系统"的功能需求、性能需求、接口需求和运行环境需求，作为后续系统设计、开发实现和测试验收的基准文档。预期的读者包括项目管理人员、系统架构师、开发工程师、测试工程师及项目评审人员。

### 1.2 背景

- **a. 待开发的软件系统的名称**：GitGet（GitHub Release 应用商店系统）
- **b. 本项目的任务提出者、开发者、用户**
  - 任务提出者：项目发起人/产品经理
  - 开发者：GitGet 开发团队
  - 用户：GitHub 平台的所有开发者及普通用户
- **c. 该软件系统同其他系统或其他机构的基本的相互来往关系**
  - 与 GitHub REST API v3 对接，获取仓库搜索、Release 列表、Asset 信息
  - 与 GitHub GraphQL API v4 对接，高效查询用户 Star 数据及关联信息
  - 与 GitHub OAuth API 对接，实现用户身份认证
  - 与操作系统本地文件系统交互，管理下载文件的存储

## 2 任务概述

### 2.1 目标

GitGet 的目标是构建一个面向 GitHub Release 资源的应用商店系统，提供以下核心价值：

1. **聚合发现**：将分散的 GitHub 项目 Release 聚合到统一的界面中，支持搜索和推荐
2. **智能匹配**：自动识别用户操作系统，展示兼容的 Release Asset
3. **高效下载**：提供专业的下载管理功能，包括进度显示、暂停/续传、并发控制
4. **个性化体验**：基于用户 Star 数据，推荐符合用户兴趣偏好的热门项目

该系统为独立运行的桌面应用，主要功能全部自含，无需依赖其他本地系统。

### 2.2 用户的特点

| 用户类型 | 特征 | 技术能力 | 使用频度 |
|----------|------|----------|----------|
| 开发者 | 熟悉 GitHub，经常下载开源工具 | 高 | 高（每天/每周） |
| 技术爱好者 | 关注新技术，喜欢尝试新软件 | 中高 | 中（每周/每月） |
| 普通用户 | 通过应用商店寻找需安装的软件 | 低中 | 低（偶尔） |

系统界面应简洁直观，降低非技术用户的学习成本，同时为高级用户提供快捷键和自定义选项。

### 2.3 假定和约束

- **开发期限**：2026年6月20日前完成核心功能
- **经费限制**：按可行性研究报告中的预算执行
- **开发人员**：团队 3-5 人（C# .NET 桌面开发、UI 设计、测试）
- **技术约束**：使用 C# + .NET 9 + Avalonia UI 构建跨平台桌面应用
- **UI 架构**：采用 MVVM 模式（CommunityToolkit.Mvvm）
- **平台兼容**：需支持 Windows 10+/macOS 11+/Ubuntu 20.04+
- **开源协议**：项目采用 MIT 协议

## 3 需求规定

### 3.1 对功能的规定

#### 功能模块总览

| 编号 | 功能模块 | 优先级 | 描述 |
|------|----------|--------|------|
| F-01 | GitHub OAuth 登录 | 高 | 用户通过 GitHub 账号授权登录 |
| F-02 | 热门项目推荐 | 高 | 基于用户 Star 类别的个性化推荐 |
| F-03 | 项目搜索 | 高 | 搜索 GitHub 公开仓库 |
| F-04 | 项目详情查看 | 高 | 展示仓库基本信息及 Release 列表 |
| F-05 | Release 智能匹配 | 高 | 自动匹配当前操作系统的 Asset |
| F-06 | 文件下载管理 | 高 | 带进度条的下载，支持暂停/续传 |
| F-07 | 用户个人中心 | 中 | 查看个人信息、Star 列表、下载历史 |
| F-08 | 系统设置 | 中 | 下载路径、主题、语言等设置 |
| F-09 | 离线缓存 | 中 | 缓存已浏览的项目信息减少 API 调用 |

#### F-01：GitHub OAuth 登录

| 项目 | 内容 |
|------|------|
| 输入 | 用户点击"登录 GitHub"按钮 |
| 处理流程 | 1. 应用打开系统浏览器跳转到 GitHub OAuth 授权页面<br>2. 启动本地 HTTP 侦听器接收回调<br>3. 用户确认授权后 GitHub 重定向到本地回调 URI<br>4. 应用用授权码换取 Access Token（HTTP POST）<br>5. 使用 Token 调用 GitHub User API 获取用户信息<br>6. Token 加密（AES-256-GCM）存入本地 SQLite |
| 输出 | 用户头像、用户名、登录状态更新到 UI |
| 约束 | Token 加密存储；支持 Token 主动撤销（登出时） |

#### F-02：热门项目推荐

| 项目 | 内容 |
|------|------|
| 输入 | 用户的 Star 列表（已登录）或默认热门（未登录） |
| 处理流程 | 1. 获取用户 Star 的仓库列表（GraphQL 批量查询）<br>2. 提取各仓库的主要编程语言和主题标签<br>3. 统计用户兴趣分类权重<br>4. 调用 GitHub Search API 查询匹配类别的热门仓库<br>5. 按 Star 数/热度排序展示 |
| 输出 | 项目卡片列表（含名称、描述、Star 数、主语言、最近更新） |
| 约束 | 未登录用户展示全局 Trending；已登录用户展示个性化推荐 |

#### F-03：项目搜索

| 项目 | 内容 |
|------|------|
| 输入 | 搜索关键词（字符串），可选筛选条件（语言、排序方式） |
| 处理流程 | 1. 调用 GitHub Search/Repositories API<br>2. 解析返回结果<br>3. 按相关度/Star 数排序展示 |
| 输出 | 搜索结果列表（分页展示） |
| 约束 | 搜索响应时间不超过 2 秒；支持中英文关键词 |

#### F-04：项目详情查看

| 项目 | 内容 |
|------|------|
| 输入 | 用户点击某项目卡片 |
| 处理流程 | 1. 获取仓库详细信息（描述、主页、Star/Fork/Watch 数、许可证）<br>2. 获取 Releases 列表（按版本倒序）<br>3. 获取各 Release 的 Asset 列表 |
| 输出 | 项目详情页（信息展示区 + Release 列表 + Asset 列表） |

#### F-05：Release 智能匹配

| 项目 | 内容 |
|------|------|
| 输入 | 用户操作系统信息（通过 Environment.OSVersion 自动检测） |
| 处理流程 | 1. 检测当前 OS 类型和架构（Windows x64/arm64、macOS x64/arm64、Linux x64/arm64）<br>2. 遍历每个 Release 的 Asset 列表<br>3. 根据文件名关键词匹配规则匹配当前 OS<br>4. 标记匹配的 Asset（如在 Windows 下匹配 .exe/.msi） |
| 输出 | 每个 Release 中标记"推荐下载"的 Asset |

**匹配规则示例：**
- Windows: .exe, .msi, .zip（含 win/windows 关键词）
- macOS: .dmg, .pkg, .app（含 mac/darwin 关键词）
- Linux: .deb, .rpm, .AppImage, .tar.gz（含 linux/ubuntu/debian 关键词）

#### F-06：文件下载管理

| 项目 | 内容 |
|------|------|
| 输入 | 用户点击某 Asset 的下载按钮 |
| 处理流程 | 1. 创建下载任务（记录 URL、文件名、目标路径、文件大小）<br>2. 通过 HttpClient.SendAsync 发起流式 GET 请求<br>3. 利用 HttpResponseMessage.Content.CopyToAsync 配合进度报告<br>4. 实时计算下载进度（已下载字节数 / 总字节数）<br>5. 更新 UI 进度条、下载速度、预计剩余时间<br>6. 支持暂停（取消 CancellationToken）/继续（Range 请求）/取消<br>7. 下载完成后触发桌面通知 |
| 输出 | 实时进度信息（百分比、速度、剩余时间）、完成通知 |
| 约束 | 支持断点续传（服务器支持 Range 请求）；最大并发下载数可配置（默认 3） |

#### F-07：用户个人中心

| 项目 | 内容 |
|------|------|
| 功能 | 查看 GitHub 用户头像、用户名、邮箱<br>查看已 Star 仓库列表<br>查看下载历史记录（已完成、进行中、已取消）<br>退出登录 |

#### F-08：系统设置

| 项目 | 内容 |
|------|------|
| 功能 | 默认下载路径设置<br>最大并发下载数设置<br>主题切换（亮色/暗色/跟随系统）<br>语言设置（中文/英文）<br>代理设置 |
| 约束 | 设置保存到 JSON 配置文件（用户目录下） |

#### F-09：离线缓存

| 项目 | 内容 |
|------|------|
| 功能 | 缓存已查看的项目基本信息（有效期 1 小时）<br>缓存搜索结果（有效期 10 分钟）<br>缓存用户 Star 列表（有效期 6 小时） |
| 约束 | 缓存数据存储在本地 SQLite 数据库 |

### 3.2 对性能的规定

#### 3.2.1 精度

- 下载进度百分比精确到 0.1%
- 下载速度显示精确到 KB/s 或 MB/s
- 文件大小显示精确到 KB、MB、GB（自动换算）
- API 数据的数值型字段保持原始精度

#### 3.2.2 时间特性要求

| 指标 | 要求 |
|------|------|
| a. 页面响应时间 | 首屏加载不超过 3 秒（含缓存情况） |
| b. 搜索响应时间 | 不超过 2 秒（网络正常情况下） |
| c. Release 列表加载 | 不超过 3 秒 |
| d. 下载响应时间 | 点击下载到开始传输不超过 1 秒 |
| e. 进度更新频率 | 每秒至少更新 1 次 |
| f. OAuth 登录流程 | 授权回调后 3 秒内完成 |

#### 3.2.3 灵活性

| 变化类型 | 适应能力 |
|----------|----------|
| a. 操作方式变化 | 同时支持鼠标和键盘快捷键操作 |
| b. 运行环境变化 | 适配 Windows/macOS/Linux 三个平台 |
| c. 接口变化 | GitHub API 版本升级时，保持向后兼容至少一个主要版本 |
| d. 精度变化 | 文件大小显示可配置精度 |
| e. 功能扩展 | 采用依赖注入 + 接口抽象设计，便于未来增加功能模块 |

### 3.3 输入输出要求

#### 输入数据

| 数据类型 | 媒体 | 格式 | 范围/约束 |
|----------|------|------|-----------|
| 搜索关键词 | 键盘输入 | UTF-8 字符串 | 1-200 字符 |
| OAuth 授权码 | 本地 HTTP 回调 | 字符串 | GitHub 自动生成 |
| 下载目标路径 | 系统文件选择对话框 | 文件系统路径 | 可写目录 |
| 软件设置 | 界面输入 | 键值对 | 见系统设置定义 |

#### 输出数据

| 数据类型 | 媒体 | 格式 | 说明 |
|----------|------|------|------|
| 项目卡片 | 屏幕显示 | 富文本卡片（Avalonia DataTemplate） | 含名称、描述、Star 数、语言等 |
| Release 列表 | 屏幕显示 | 结构化列表（ItemsControl） | 版本号、日期、Asset 列表 |
| 下载进度 | 屏幕显示 | 可视化进度条（ProgressBar） | 含百分比、速度、剩余时间 |
| 通知消息 | 系统通知 | 文本 + 图标 | 下载完成、错误等通知 |
| 下载文件 | 本地磁盘 | 原始文件格式 | 保存在用户指定目录 |

### 3.4 数据管理能力要求

| 数据类型 | 预估数据量 | 存储方式 |
|----------|-----------|----------|
| 用户 Token | 每用户一条 | AES-256-GCM 加密存储于 SQLite |
| 项目缓存 | 最多 500 条 | SQLite 本地数据库 |
| 搜索缓存 | 最多 200 条 | SQLite 本地数据库 |
| 下载历史 | 最多 1000 条 | SQLite 本地数据库 |
| 用户设置 | 约 50 个键值对 | JSON 配置文件 |

首年预估总数据存储需求不超过 50MB。

### 3.5 故障处理要求

| 故障类型 | 表现 | 处理措施 |
|----------|------|----------|
| 网络中断 | API 请求失败、下载中断 | 自动重试（最多 3 次），显示友好错误提示；已下载部分保留，网络恢复后通过 Range 请求自动继续 |
| API 速率限制 | 返回 403 或 429 状态码 | 显示剩余配额，建议用户登录以提高配额；自动排队等待配额恢复（HTTP Retry-After 头） |
| OAuth 令牌过期 | API 返回 401 | 清除过期 Token，要求用户重新登录 |
| 文件写入失败 | 磁盘空间不足、权限不足 | 提示用户选择其他路径 |
| 应用崩溃 | 应用异常退出 | 下次启动时从 SQLite 读取未完成任务状态，基于断点续传恢复 |
| GitHub 服务异常 | API 响应缓慢或 5xx | 显示服务异常提示，建议稍后重试 |

### 3.6 其他专门要求

| 要求类别 | 具体要求 |
|----------|----------|
| 安全保密 | Token 加密存储（AES-256-GCM）；HTTPS 通信；不记录明文的用户凭据 |
| 易用性 | 首次启动有引导教程；主界面符合应用商店布局习惯；操作反馈即时 |
| 可维护性 | 代码模块化，遵循 .NET 命名规范和 StyleCop 规则；关键功能有单元测试覆盖（>60%） |
| 可扩展性 | 推荐算法通过 IRecommendationStrategy 接口抽象；UI 通过 Avalonia 的 ControlTemplate 可定制 |
| 可靠性 | 正常使用条件下连续运行 72 小时无崩溃；API 请求失败有降级策略 |
| 可访问性 | 支持键盘 Tab 导航；支持屏幕缩放；Avalonia 原生支持无障碍（AutomationPeer） |

### 4.3 接口

| 接口 | 协议 | 说明 |
|------|------|------|
| GitHub REST API | HTTPS / REST | 仓库搜索、Release 列表、用户信息 |
| GitHub GraphQL API | HTTPS / GraphQL | 高效的 Star 数据查询 |
| GitHub OAuth API | HTTPS / OAuth 2.0 | 用户身份认证 |
| 操作系统文件系统 | .NET System.IO API | 文件读写管理 |
| 系统通知 | Avalonia 原生通知 API | 下载完成等通知推送 |

### 4.4 控制

- **启动**：用户双击应用图标启动，自动检查更新
- **退出**：用户关闭窗口时，正在进行的下载任务提示用户（继续后台/取消）
- **后台运行**：支持最小化到系统托盘，后台继续下载任务
- **自动更新**：应用启动时自动检查新版本，可选更新
- **资源控制**：当下载并发数达到上限时，新的下载任务进入排队等待状态
